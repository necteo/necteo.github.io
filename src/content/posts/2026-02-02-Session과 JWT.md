### 🔹 세션(Session) vs JWT

### ✅ 왜 JWT를 쓰게 되었나요?

> 프론트엔드(React)와 백엔드(Spring Boot)가 분리된 구조에서
> 세션 기반 인증은 서버 상태를 관리해야 해서 확장성과 관리가 어려웠고,
> 이를 해결하기 위해 Stateless한 토큰 기반 인증(JWT) 이 널리 사용되게 되었습니다.

### 🔹 Session 인증

#### 📌 개념

- 로그인 시 서버가 세션 생성

- 세션 ID를 쿠키로 클라이언트에 전달

- 이후 요청마다 세션 ID로 사용자 식별

#### 👍 장점

- 구현이 단순

- 서버에서 인증 정보 직접 관리 → 보안 제어 용이

- 로그아웃, 세션 만료 처리 쉬움

#### 👎 단점

- 서버에 상태 저장 → 확장성 낮음

- 서버 분산 시 Redis 등 추가 인프라 필요

- 프론트/백엔드 분리 + CORS 환경에서 설정 복잡

### 🔹 JWT 인증

#### 📌 개념

- 로그인 시 서버가 JWT 발급

- 클라이언트가 토큰 저장

- 요청마다 Authorization: Bearer JWT 전달

- 서버는 토큰 검증만 수행 (Stateless)

#### 👍 장점

- 서버에 상태 저장 안 함 → 확장성 좋음

- 프론트/백엔드 분리 구조에 적합

- 모바일, 외부 API, MSA 환경에 유리

#### 👎 단점

- 토큰 탈취 시 위험

- 서버에서 즉시 토큰 무효화 어려움

- 토큰 크기 큼 → 네트워크 비용 증가

### 🔹 핵심 차이 한 눈에 보기

| 구분        | Session                  | JWT                   |
| ----------- | ------------------------ | --------------------- |
| 상태        | 저장                     | 서버 없음 (Stateless) |
| 확장성      | 낮음                     | 높음                  |
| 보안 통제   | 쉬움                     | 상대적으로 어려움     |
| 로그아웃    | 쉬움                     | 별도 처리 필요        |
| 적합한 환경 | 단일 서버, 관리자 페이지 | SPA, 모바일, MSA      |

### 🔹 실무에서는?

> Access Token(JWT) + Refresh Token(DB/Redis) 구조를 가장 많이 사용합니다.

- Access Token: 짧은 만료 (10~30분)

- Refresh Token: 서버 저장, 탈취 시 제어 가능

- 보안 + 확장성 절충

### 🔹 면접 한 방 답변 (암기용)

> "프론트와 백엔드가 분리된 환경에서는 세션 기반 인증이 서버 확장성과 관리 측면에서 불리해서,
> 서버 상태를 저장하지 않는 JWT 기반 인증이 많이 사용됩니다.
> 다만 JWT는 토큰 무효화가 어렵기 때문에, 실무에서는 Refresh Token을 함께 사용하는 하이브리드 방식을 주로 사용합니다."
